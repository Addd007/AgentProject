# docs 目录规范

本目录用于承载面向团队协作、部署交接、系统演进的正式文档，目标是把原本分散在 README、临时笔记和口头说明中的信息沉淀为可维护的项目文档体系。

## 1. 目录职责

建议后续按以下分层维护文档：

```text
docs/
├─ README.md                文档导航与编写规范
├─ architecture/            架构设计、时序图、模块关系
├─ api/                     接口说明、鉴权方式、示例请求
├─ deployment/              本地、测试、生产部署说明
├─ operations/              运维手册、备份恢复、故障处理
├─ development/             开发规范、调试方法、提交流程
└─ product/                 业务背景、使用场景、术语说明
```

当前仓库可以先按需逐步创建子目录，不需要一次性补齐全部内容，但新增文档时应优先放入 docs，而不是继续堆在仓库根目录。

## 2. 与 README 的边界

README 负责回答三个问题：

- 这是什么项目。
- 如何快速启动。
- 核心能力和整体架构是什么。

docs 负责承载更长、更细、更适合协作和交付的内容，例如：

- API 详细字段说明
- 架构设计决策
- 数据库表设计
- 生产部署步骤
- 运维排障手册
- 开发规范和提交流程

简单说：README 面向快速理解，docs 面向持续协作。

## 3. 命名规范

- 文件名统一使用小写英文和中划线，例如 api-auth.md、session-storage.md。
- 尽量避免中文文件名、空格和过长名称。
- 一篇文档只解决一个主题，避免把架构、部署、排障混在同一个文件里。

## 4. 编写规范

- 先写结论，再写细节，优先保证可扫读性。
- 每篇文档开头建议包含：背景、适用范围、前置条件。
- 涉及命令时给出可直接执行的示例。
- 涉及配置时明确变量来源、默认值、风险点。
- 涉及系统行为时尽量引用真实代码路径，而不是抽象描述。
- 涉及生产操作时必须标明影响范围和回滚方式。

## 5. 建议优先补齐的文档

结合当前项目现状，建议优先新增以下文档：

1. architecture/system-overview.md
2. api/backend-endpoints.md
3. deployment/local-development.md
4. deployment/production-deployment.md
5. operations/backup-and-recovery.md
6. development/contribution-guide.md

## 6. 维护原则

- 新增功能时，同时更新对应文档。
- 目录级文档优先链接已有文档，避免内容重复复制。
- 过期文档应及时删除或标记 obsolete，避免多个版本并存。
- 文档描述必须以当前仓库实现为准，不写“计划中”内容冒充现状。

如果后续继续整理文档，建议先从架构、API 和部署三类开始，这三类对新同事接手和项目交付最关键。
